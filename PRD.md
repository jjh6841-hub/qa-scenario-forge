# Product Requirements Document (PRD)
## QA Scenario Forge - 의료 EMR 특화 AI QA 도구

**버전**: 1.0.0
**작성일**: 2026-03-13
**상태**: In Development

---

## 1. 문제 정의 (Problem Statement)

의료 EMR(전자의무기록) 소프트웨어는 환자 안전과 직결되고, 심평원 규정·의료법·개인정보보호법 등 복잡한 법령 준수 요구사항이 있어 다른 소프트웨어보다 훨씬 높은 QA 품질이 요구된다.

그러나 현재 EMR 소프트웨어 QA 현장에서는:
- **규정 도메인 지식 부족**: QA 엔지니어가 의료 법령과 심평원 규정을 충분히 숙지하지 못해 중요한 컴플라이언스 리스크를 놓치는 경우가 많다
- **반복적인 수작업**: 기능 명세서에서 테스트 케이스를 수작업으로 작성하는 데 많은 시간이 소요된다
- **테스트 자동화 코드 작성 어려움**: Playwright 등 자동화 도구를 활용한 테스트 코드 작성에 전문 지식이 필요하다
- **리스크 기반 테스트 부재**: 중요도와 발생 가능성에 따른 체계적인 리스크 평가 없이 테스트가 진행된다

## 2. 대상 사용자 (Target Users)

### 주요 사용자
- **의료 소프트웨어 QA 엔지니어**: EMR, PACS, OCS 등 의료 SW를 테스트하는 QA 담당자
- **의료 소프트웨어 개발자**: 기능 개발 후 자체 테스트를 수행하는 개발자
- **QA 매니저**: 테스트 전략 수립 및 리스크 평가를 담당하는 관리자

### 사용자 특성
- 한국 의료 도메인 지식 보유
- QA 경험 있으나 자동화 코드 작성 경험 미숙할 수 있음
- 빠른 테스트 케이스 산출물 필요

## 3. 기능 요구사항 (Feature Specifications)

### Feature 1: 기능 명세서 입력
**우선순위**: P0 (핵심 기능)

- 사용자는 자유 형식의 한국어 기능 명세서를 텍스트 영역에 입력할 수 있다
- 세 가지 EMR 도메인 프리셋 제공 (처방 입력, 로그인/인증, 진료 기록 조회)
- 프리셋 클릭 시 해당 도메인의 상세 명세서가 자동 입력됨
- 입력된 텍스트는 전역 상태로 관리되어 파이프라인 전체에서 사용됨

**수용 기준**:
- 텍스트 입력 시 실시간으로 글자 수가 표시된다
- 프리셋 선택 시 활성 상태가 시각적으로 표시된다
- 입력 내용이 없을 때 분석 버튼이 비활성화된다

### Feature 2: AI 리스크 분석 (Stage 1)
**우선순위**: P0

- Claude AI가 기능 명세서를 분석하여 잠재적 리스크를 식별한다
- 리스크 카테고리: 환자 안전, 규정 준수, 데이터 무결성, 보안, 성능, 사용성
- 각 리스크는 영향도(1-5), 발생가능성(1-5), 우선순위(critical/high/medium/low) 포함
- 심평원 DUR 규정, 의료법, 개인정보보호법 관련 리스크 특화 분석
- 결과는 5x5 히트맵과 정렬된 리스크 목록으로 시각화

**수용 기준**:
- 최소 8개 이상의 리스크가 식별된다
- 리스크 히트맵에서 영향도/발생가능성 좌표에 올바르게 표시된다
- 우선순위에 따라 색상 코딩이 적용된다

### Feature 3: 테스트 시나리오 생성 (Stage 2)
**우선순위**: P0

- 식별된 리스크를 기반으로 테스트 시나리오를 생성한다
- 시나리오 유형: functional, security, compliance, performance, usability
- 각 시나리오는 사전 조건, 상세 설명, 관련 리스크 ID 포함
- 실제 의료 현장 워크플로우를 반영한 현실적인 시나리오

**수용 기준**:
- 최소 6개 이상의 시나리오가 생성된다
- 각 시나리오가 하나 이상의 리스크와 연결된다
- 유형별 색상 배지로 구분된다

### Feature 4: 상세 테스트 케이스 생성 (Stage 3)
**우선순위**: P0

- 시나리오를 기반으로 단계별 테스트 케이스를 작성한다
- 각 단계는 구체적인 액션과 기대 결과를 포함한다
- 우선순위(P1/P2/P3) 및 자동화 가능 여부 포함
- 클릭으로 단계를 펼쳐 볼 수 있는 인터랙티브 테이블

**수용 기준**:
- 최소 10개 이상의 테스트 케이스가 생성된다
- 각 케이스가 3-7개의 단계를 포함한다
- 자동화 가능 케이스와 수동 케이스가 구분된다

### Feature 5: Playwright 자동화 코드 생성 (Stage 4)
**우선순위**: P1

- 자동화 가능한 테스트 케이스를 Playwright TypeScript 코드로 변환한다
- Page Object Model 패턴 적용
- 실행 가능한 완전한 테스트 파일 생성
- 파일 탭 전환, 구문 강조, 클립보드 복사 기능 제공

**수용 기준**:
- 최소 2개의 테스트 파일이 생성된다
- vscDarkPlus 테마의 구문 강조가 적용된다
- 클립보드 복사 버튼이 동작한다

### Feature 6: 결과 내보내기
**우선순위**: P1

- 분석 결과를 JSON 또는 Markdown 형식으로 다운로드한다
- JSON: 전체 분석 데이터 구조적으로 저장
- Markdown: 사람이 읽기 쉬운 형식으로 전체 보고서 생성
- 파일명에 날짜가 자동 포함됨

**수용 기준**:
- 분석 결과가 있을 때만 내보내기 버튼이 표시된다
- JSON 파일이 올바른 구조로 생성된다
- Markdown 파일에 모든 섹션이 포함된다

## 4. 비기능 요구사항 (Non-Functional Requirements)

### 성능
- 초기 페이지 로드: 3초 이내
- 각 AI 분석 단계: 30초 이내 응답 (Claude API 의존)
- 4단계 전체 파이프라인: 2분 이내 완료

### 접근성 및 사용성
- 전체 한국어 인터페이스
- 다크 테마 전용 UI
- 반응형 레이아웃 (최소 1280px 권장)
- 스크린 리더 지원을 위한 aria 속성

### 보안
- API 키는 환경 변수로만 관리
- 민감한 의료 데이터는 브라우저 메모리에만 저장 (서버 미전송)
- HTTPS 배포 필수

### 브라우저 지원
- Chrome 120+
- Edge 120+
- Firefox 120+

## 5. 기술 아키텍처 (Tech Architecture)

```
Frontend (SPA)
├── Vite 8 + React 19 + TypeScript 5.9
├── Tailwind CSS v4 (스타일링)
├── React Context + useReducer (상태 관리)
└── Anthropic SDK @anthropic-ai/sdk (AI 연동)

Testing
├── Vitest + React Testing Library (단위/통합)
├── Playwright (E2E)
└── @vitest/coverage-v8 (커버리지)

Deployment
├── Vercel (SPA 호스팅, SPA rewrites)
└── GitHub Actions (CI/CD)
```

## 6. Out of Scope

- **백엔드 서버**: 현재 버전은 순수 프론트엔드 SPA. API 키 보호를 위한 프록시 서버 없음
- **사용자 인증**: 로그인/계정 기능 없음
- **데이터 영속성**: 분석 결과의 DB 저장 없음 (새로고침 시 초기화)
- **협업 기능**: 결과 공유, 댓글, 리뷰 기능 없음
- **다국어 지원**: 한국어 전용 (영어 인터페이스 미지원)
- **오프라인 모드**: Claude API 연결 필수
- **모바일 최적화**: 태블릿/모바일 레이아웃 최적화 미포함
